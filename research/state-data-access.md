# State data access: what exists, what to ask for

Researched 2026-07-02, prompted by the question: could Coverage Compass get a privileged level of access to state data, to know a member's actual care status and to know exactly which letter a member received? Method matched the prior-art survey: parallel web research against primary sources, then an adversarial verification pass that re-fetched every load-bearing claim (hcpf.colorado.gov serves documents to a browser user agent even where it bot-blocks naive fetches; hhs.gov and medicaid.gov block everything and are noted where it matters).

The headline: **"privileged access" is mostly the wrong frame. Colorado already has a member-authorized data lane, live today, and the letters already identify themselves.** The right asks are consent-based and cheap for the state. The companion proposal is [`docs/proposal-hcpf-data-access.md`](../docs/proposal-hcpf-data-access.md).

## 1. The member-authorized lane (live today)

Federal rules (CMS-9115-F, 2020) require state Medicaid fee-for-service programs to run a FHIR Patient Access API: the member authorizes a third-party app of their choice, and the app retrieves that member's data. Colorado complies:

- **Health First Colorado's Patient Access API is live.** The state's own member page lists three working consumer apps: b.well, OneRecord, and Flexpa ([healthfirstcolorado.gov](https://www.healthfirstcolorado.gov/news-resources/accessing-your-health-claims-data/)).
- **Verified live resources: Patient, Coverage, and ExplanationOfBenefit** ([Flexpa's endpoint directory](https://www.flexpa.com/docs/network/directory/0cc16dff-1527-4074-8329-1d725b64b62d), updated 2026-05-28). Coverage means **enrollment status is queryable with member consent**, which is precisely the "care status" capability in question. Tokens are refreshable with an indefinite authorization period.
- **Who runs it:** Onyx Technology's SAFHIR platform (the `co-prd` tenant is active in [SAFHIR's own registry](https://api-dmdh-prd.safhir.io/registration-portal/portal/data-providers/list)). Registration is vendor-mediated: sandbox at portal.safhir.io, then email support@safhir.io for production tenant access. Gainwell holds the "CMS Interoperability and Patient Access" module in HCPF's MMIS per [HCPF's procurement page](https://hcpf.colorado.gov/mmis-procurement).
- **History that matters for durability:** CMS's official endpoint directory records a first go-live around January 2024 on a 1upHealth deployment (api.coloradofhir.com). That domain is now dead (DNS NXDOMAIN, verified 2026-07-02); the live implementation is SAFHIR. Colorado has already changed interoperability vendors once, and **CMS's public directory still points at the dead endpoint**. HCPF publishes no developer page at all. Both are citable gaps a proposal can politely surface.
- **Precedents:** Utah Medicaid [publicly lists](https://medicaid.utah.gov/interoperability/) b.well and Flexpa as registered apps; Connecticut DSS [registers apps through SAFHIR](https://portal.ct.gov/DSS/ITS/DSS-HealthIT/Business-Intelligence-and-DSS-HealthIT/Interoperability); and Colorado's own member page names three consumer apps. A nonprofit app registering is not a novel ask.
- **CMS-0057-F** adds prior-auth data to the Patient Access API and three more APIs, with state Medicaid FFS compliance "primarily January 1, 2027" ([89 FR 8758](https://www.federalregister.gov/documents/2024/02/08/2024-00895/medicare-and-medicaid-programs-patient-protection-and-affordable-care-act-advancing-interoperability); [HCPF's 0057 page](https://hcpf.colorado.gov/cms-interoperability-and-prior-authorization-final-rule-cms-0057-f)).

Two verified problems to design around:

1. **SAFHIR's developer docs describe only confidential OAuth clients** ("configured as Confidential Clients ... securely maintain credentials"). Public, secretless browser clients (which is what a no-server app is) are not documented, even though the SMART App Launch spec explicitly sanctions them (PKCE S256, CORS). This is the single biggest architecture question and must be asked directly of Onyx/HCPF. If only confidential clients are allowed, the choices are a minimal token-exchange relay (a real change to our zero-server posture that would need its own privacy analysis) or advocating for public-client support.
2. **The connection is unreliable in practice:** Flexpa measures an **11.18% authorization success rate** for the Colorado Medicaid connection over the last six months. For a disabled population, a login flow that fails nine times in ten is itself an accessibility failure, and worth raising as a member-experience issue independent of our app.

## 2. The partner lane (real, but organizationally gated)

- **PEAK Pro** is PEAK's partner tier (current version launched 2024-03-01 for Case Management Agencies: financial eligibility checks and LTSS Level of Care status; [HCPF OM 24-002](https://hcpf.colorado.gov/sites/hcpf/files/HCPF%20OM%2024-002%20Launch%20of%20Streamline%20Eligibility%20&%20PEAKPro.pdf)). HCPF's [3rd Party System User Access Request form](https://hcpf.colorado.gov/sites/hcpf/files/HCPF%203rd%20Party%20-%20System%20User%20Access%20Request%2008.07.2024.pdf) explicitly lists **"Community Based Organization (CBO)"** as a PEAK Pro user type, with a Read Only option. The gate is organizational: the org must exist in PEAK Pro, each user signs a System User Agreement with minimum-necessary justification, and an **HCPF contract/program manager signs**, implying a formal relationship, not just goodwill.
- **The assistance-site ladder:** Certified Application Assistance Site (nonprofit; training plus a 95% test; no system access) -> after 90 days and 50+ clients/quarter, eligible to apply as a **Medical Assistance site** (CRS 25.5-4-205), which gets real CBMS inquiry access: eligibility, redetermination, eligibility spans, case comments, PEAK inbox. This is the durable path for CCDC advocates, independent of our software.
- **Authorized Representative** designation (10 CCR 2505-10 8.057; addable directly on PEAK) lets a trusted person or organization see the member's information and act on coverage issues, but nothing found grants an AR independent portal access to the member's case.
- Poor fits, checked and set aside: Contexture (the HIE) is clinical-only with no CBO lane; the enrollment broker handles plan choice only; person-level data releases outside contracts have no found precedent (BIDM/COGNOS is contractor-only).

## 3. Knowing what the letter is (the surprising win)

- **CBMS letters identify themselves.** HCPF's published [sample MAGI renewal packet](https://hcpf.colorado.gov/sites/hcpf/files/Magi%20English%20Standard%20October%202023.pdf) carries a template identifier in the footer of every page (`Med_MAGI_Redetermination_Notice12_EN`) plus a `Case Number/Correspondence ID` line. Verified page by page. **Our classifier can read an exact template ID on-device today**; what is missing is the lookup table from template ID to meaning.
- **That table exists and is small.** The 2023 State Auditor performance audit ([report 2261P](https://leg.colorado.gov/sites/default/files/documents/audits/2261p_medicaid_correspondence.pdf)) documents 143 identified CBMS letter templates, 80 retired, 4 added: **63 active templates** subject to SB 17-121's plain-language requirements. The audit also found HCPF "did not have a comprehensive or updated inventory of letters" at the start, that CBMS sends 400,000+ eligibility letters a month, and that **90% of 80 reviewed letters had at least one problem**. The inventory is a specific, identifiable record; HCPF has a CORA process if a friendly ask fails.
- **The H.R.1 notices are being drafted right now, in public.** HCPF's quarterly [Member Correspondence Improvements](https://hcpf.colorado.gov/member-correspondence-improvements) stakeholder process reviewed "Worksheet J: Work Requirements Form" in February 2026 and a draft expansion-population notice in April 2026 (feedback: hcpf_stakeholders@state.co.us). SB 17-121 requires HCPF to get advocate feedback before significant correspondence changes, so CCDC belongs in that room regardless of this project. A [sample work-requirements notice dated 2026-08-01](https://hcpf.colorado.gov/sites/hcpf/files/Work%20Requirements%20Notice%20(Narrow).pdf) is downloadable from HCPF today and is usable as a classifier fixture immediately.
- The LTSS notice taxonomy (LTC NOA and friends) is documented in [HCPF OM 25-027](https://hcpf.colorado.gov/sites/hcpf/files/HCPF%20OM%2025-027%20Notice%20of%20Action%20Requirements%20for%20Case%20Management%20Agencies.pdf).

## 4. Legal posture

- **HIPAA:** under HHS OCR guidance (FAQ 3009 and the access-rights/apps guidance; hhs.gov bot-blocks, but Denver Health Medical Plan [states the rule verbatim](https://www.denverhealthmedicalplan.org/interoperability-health-information-and-third-party-apps) to its Colorado members), once a member directs their data to an app of their choosing that is neither a covered entity nor a business associate, HIPAA no longer applies downstream and the payer bears no liability. FTC rules govern the app instead. Our posture is stronger than the typical app's: we never receive the data server-side at all.
- **Denial standard:** a payer may deny an app's API registration only for a determined security threat to PHI on the payer's own systems; privacy attestations may be requested, not required ([1upHealth's developer docs](https://docs.1up.health/docs/get-started/third-party-apps) state the CMS standard), though platform listing practices can add friction in practice.
- **Information blocking** does not automatically reach payers ([healthit.gov FAQ](https://www.healthit.gov/faq/are-health-plans-or-other-payers-subject-information-blocking-regulation)); the CMS API rules are the lever.
- **Colorado Privacy Act** covers nonprofits above 100k consumers/year with only a data-level HIPAA exemption; a client-side app that never receives data server-side largely sidesteps it, but this needs counsel review alongside the Terms work.

## 5. The H.R.1 verification context (why the state should care)

- CMS-2454-IFC (published 2026-06-03, effective 2026-07-31, 91 FR 33348) requires implementation by 2027-01-01 and requires states to check data ex parte before demanding paperwork. Colorado's IAPD builds a CBMS MVP (Deloitte) with self-attestation forms and iOCR/AI scanning; **work-requirement letters start going to members in August 2026** ([HCPF FAQ](https://hcpf.colorado.gov/work-requirements-faqs)).
- The ex parte sources (SWICA wage records, Equifax's The Work Number, NDNH, SNAP/TANF data, claims) systematically miss gig work, self-employment, caregiving, and volunteering ([SHVS brief](https://shvs.org/wp-content/uploads/2025/09/Coverage-and-Access_Work-Requirements-Compliance-and-Exeptions-Verification_09.12.2025_Rev.pdf)), exactly the cases that fall to member self-attestation, which is the population our tool serves. Equifax's pricing leverage over states (NC: +24% in 2022, +36% in 2024) is a live policy concern.
- CMS itself is building **Emmy ("Eligibility Made Easy")**, an open-source (CC0) consent-based verification prototype ([iv-cbv-payroll](https://github.com/DSACMS/iv-cbv-payroll), [emmy-api](https://github.com/CMSgov/emmy-api)). Consent-based member-facing verification is the direction the federal government is already funding, which makes our ask legible rather than exotic.
- Arkansas 2018 remains the argument: 18,000 lost coverage in seven months, most unaware until they sought care; 904 good-cause requests and 69 appeals total ([KFF](https://www.kff.org/medicaid/issue-brief/state-data-for-medicaid-work-requirements-in-arkansas/)).

## 6. What we should NOT ask for

Bulk or backend access to member data; anything requiring us to hold PII server-side (it breaks the product's core promise); credential linking (Propel's EBT model works at 5M households/month scale but is the anti-pattern OAuth exists to replace); anything that requires new rulemaking before the January 2027 wave.
